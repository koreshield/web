# Troubleshooting

Solutions for common issues encountered when integrating or operating Koreshield.

---

## Requests blocked incorrectly (false positives)

**Symptom:** Legitimate user prompts are being flagged and blocked by Koreshield.

**Solutions:**

1. Review blocked requests in **Dashboard → Threat Monitoring** to identify the patterns being flagged.
2. Adjust the detection confidence threshold in your policy configuration:
   ```yaml
   policies:
     prompt_injection:
       confidence_threshold: 0.85  # raise to reduce false positives
   ```
3. Create allowlist rules for specific patterns that are safe in your context.
4. If false positives persist on a particular attack family, contact support - detection tuning may be needed for your traffic profile.

---

## High latency on scanned requests

**Symptom:** Requests routed through Koreshield are noticeably slower than direct provider calls.

**Solutions:**

1. Koreshield adds sub-30ms overhead at the proxy layer. If latency is higher, check:
   - Network path between your app and the Koreshield proxy.
   - Provider health on **Dashboard → Provider Health** - slow providers will appear as degraded.
2. For self-hosted deployments, ensure the proxy container has sufficient CPU and memory.
3. RAG document scans add latency proportional to document count. If scanning large retrieval contexts, consider reducing the number of documents passed per request.

---

## Provider health shows "degraded"

**Symptom:** The dashboard or `/health` endpoint shows one or more providers as degraded.

**Solutions:**

1. Check the **Provider Health** page for error rates and response times per provider.
2. Verify your provider API keys are valid and have sufficient quota.
3. Check the provider's status page (e.g., [status.openai.com](https://status.openai.com)) for outages.
4. If only one provider is degraded, Koreshield continues routing to healthy providers - no action is needed unless it's your primary provider.

---

## API key authentication failures

**Symptom:** Requests return `401 Unauthorized` or `403 Forbidden`.

**Solutions:**

1. Verify the API key is correctly set in your request headers: `Authorization: Bearer ks_...`
2. Check that the key hasn't been revoked in **Dashboard → API Keys**.
3. Ensure the key has the required scopes for the endpoint you're calling.
4. For team-scoped keys, verify the key belongs to the correct team.

---

## Self-hosted: container fails to start

**Symptom:** The Koreshield Docker container exits immediately after starting.

**Solutions:**

1. Check container logs: `docker compose logs api`
2. Common causes:
   - **Missing `KS_LICENSE_KEY`**: Self-hosted deployments require a valid Enterprise licence key.
   - **Expired licence**: If the licence has expired beyond the 14-day grace period, the server refuses to start.
   - **Database connection failure**: Verify `DATABASE_URL` is set and the database is reachable.
3. Run the verification script: `python scripts/verify_installation.py`

---

## Self-hosted: licence validation errors

**Symptom:** The health endpoint returns licence errors or the server logs `licence_invalid`.

**Solutions:**

1. Verify `KS_LICENSE_KEY` is set correctly in your environment.
2. Check licence expiry with the `/health` endpoint - it includes `days_until_expiry`.
3. If the licence has expired but is within the 14-day grace period, the server will continue operating with a warning header (`X-KoreShield-Licence: grace-period`).
4. To renew, contact the Koreshield team for a new licence key.
5. Hot-reload a renewed licence by sending `SIGHUP` to the running process: `kill -HUP <pid>`

## Air-gapped: threat corpus not loading

**Symptom:** The health endpoint shows `external_entries: 0` despite having a corpus file configured.

**Solutions:**

1. Verify `KS_THREAT_CORPUS_PATH` points to an existing JSONL file.
2. Check file permissions - the container process must be able to read the file.
3. Verify the JSONL format. Each line must be valid JSON with at least `text` and `label` fields:
   ```json
   {"text": "ignore previous instructions", "label": 1, "family": "injection"}
   ```
4. Check container logs for `corpus_line_parse_error` warnings that indicate malformed lines.
5. After updating the corpus file, send `SIGHUP` to reload: `kill -HUP <pid>`

## Dashboard shows "Failed to load page"

**Symptom:** Some pages show an error message and require a manual browser refresh.

**Solutions:**

1. This is typically a stale chunk cache issue after a deployment. Hard-refresh the page (`Ctrl+Shift+R` / `Cmd+Shift+R`).
2. Koreshield includes automatic retry logic for chunk-load failures, but in rare cases a manual refresh is needed.
3. If this happens consistently, clear your browser cache.

---

## Rate limit errors (HTTP 429)

**Symptom:** Requests return `429 Too Many Requests`.

**Solutions:**

1. Check your plan's rate limit on the **Usage & Limits** page.
2. Rate limits are per-minute. If you're hitting limits during traffic spikes, consider upgrading to a higher plan.
3. For self-hosted deployments with licence-based monthly caps, the 429 indicates you've exceeded your monthly protected request allocation. Contact Koreshield to increase your cap.
4. Implement exponential backoff in your client code for transient rate limit responses.

---

## WebSocket connection drops

**Symptom:** Real-time dashboard updates stop working or WebSocket connections are refused.

**Solutions:**

1. Check that your network/proxy allows WebSocket upgrades.
2. If behind a reverse proxy (Nginx, Caddy, Cloudflare), ensure WebSocket support is enabled.
3. Koreshield's dashboard will automatically attempt to reconnect on connection drops.
