# REST API

Koreshield's REST API contract is generated from the live FastAPI app.

- Interactive docs (Swagger UI): `GET /docs`
- ReDoc: `GET /redoc`
- OpenAPI JSON: `GET /openapi.json`

## Authentication Model

Protected endpoints require authentication via one of these methods:

1. `Authorization: Bearer <jwt>`: JWT issued via the dashboard login
2. `X-API-Key: ks_...`: API key provisioned via the dashboard
3. `ks_access_token` httpOnly cookie: JWT set during login

JWT validation is strict:

- issuer (`iss`) is required and verified
- audience (`aud`) is required and verified
- expiry (`exp`) and issued-at (`iat`) are verified
- key policy enforces either `HS256` secret mode or `RS256` keypair mode

### Usage weights

Different scan types consume different numbers of protected requests against your monthly plan limit:

| Scan type | Weight | Example |
|---|---|---|
| Prompt / response (`/v1/chat/completions`, `/v1/scan`, `/v1/scan/batch`) | **1** | 100 scans = 100 requests |
| RAG document scan (`/v1/rag/scan`) | **20** | 100 scans = 2,000 requests |
| VoiceGuard audio scan (`/v1/audio/scan`) | **50** | 100 scans = 5,000 requests |

Heavier scan types require more server-side resources (multi-document parsing, speech transcription) and are weighted accordingly. Tool session auto-gating with `speech_context` does **not** consume additional audio-scan units — only explicit calls to `/v1/audio/scan` are weighted.

## `/v1/chat/completions`: Proxy Endpoint

### Streaming

Pass `"stream": true` in the JSON body to receive a `text/event-stream` (Server-Sent Events) response. Tokens arrive as OpenAI SSE chunks (`data: {...}\n\n`) and the stream ends with `data: [DONE]\n\n`. This works the same way regardless of which backend provider is selected.

```bash
curl -s -N -X POST https://api.koreshield.com/v1/chat/completions \
  -H "Authorization: Bearer <TOKEN>" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "gpt-4o",
    "messages": [{"role": "user", "content": "Hello"}],
    "stream": true
  }'
```

### Automatic Model Routing

Koreshield routes requests to the correct upstream provider based on the model name prefix, with no extra configuration needed:

| Model prefix | Provider |
|---|---|
| `gpt-*`, `o1-*`, `o3-*`, `o4-*` | OpenAI |
| `claude-*` | Anthropic |
| `gemini-*` | Google Gemini |
| `deepseek-*` | DeepSeek |

If the preferred provider is unhealthy, Koreshield automatically fails over to the next available provider.

### Blocked Request Response

When the prompt scan blocks a request, the endpoint returns **HTTP 403**:

```json
{
  "blocked": true,
  "reason": "High-confidence prompt injection detected",
  "risk_class": "high",
  "review_required": true,
  "capability_signals": ["ignore_instructions", "role_override"],
  "error": {
    "code": "prompt_injection",
    "message": "Blocked: High-confidence prompt injection detected"
  }
}
```

## `/v1/scan`: Prompt Scan Response

### Response Fields

| Field | Type | Description |
|---|---|---|
| `blocked` | boolean | Whether the prompt was blocked |
| `confidence` | float | Detection confidence from 0 to 1 |
| `attack_type` | string \| null | Primary detected attack category |
| `attack_categories` | string[] | All detected attack categories |
| `indicators` | string[] | Specific patterns that triggered detection |
| `message` | string | Human-readable scan summary |
| `request_id` | string | Unique ID for this scan (use in audit logs) |
| `severity` | string | `none` \| `low` \| `medium` \| `high` \| `critical` |
| `threat_level` | string | Detailed threat level string |
| `reason` | string \| null | Detailed block reason (if blocked) |
| `processing_time_ms` | float | Server-side processing time |
| `timestamp` | string | ISO 8601 UTC timestamp |

Returns **HTTP 403** when `blocked: true`, **HTTP 200** otherwise.

```json title="Example safe response (HTTP 200)"
{
  "blocked": false,
  "confidence": 0.04,
  "attack_type": null,
  "attack_categories": [],
  "indicators": [],
  "message": "Prompt is safe.",
  "request_id": "3e4f5a6b-...",
  "severity": "none",
  "threat_level": "none",
  "reason": null,
  "processing_time_ms": 8.2,
  "timestamp": "2026-04-12T10:30:00Z"
}
```

```json title="Example blocked response (HTTP 403)"
{
  "blocked": true,
  "confidence": 0.94,
  "attack_type": "prompt_injection",
  "attack_categories": ["prompt_injection", "system_override"],
```

## `/v1/audio/scan`: VoiceGuard Speech Scan

VoiceGuard is transcript-first. Send ASR output plus source trust metadata before speech reaches an LLM or tool chain.

```bash
curl -s -X POST https://api.koreshield.com/v1/audio/scan \
  -H "Authorization: Bearer <TOKEN>" \
  -H "Content-Type: application/json" \
  -d '{
    "transcript": "Summarise this call for the account notes.",
    "source_type": "live_microphone",
    "channel": "user_speech",
    "speaker_verified": true,
    "known_user": true,
    "policy_id": "voiceguard-default",
    "intended_use": "summarization",
    "asr": { "confidence": 0.96 },
    "tools_available": ["lookup_customer", "refund_payment"]
  }'
```

Key response fields: `decision`, `risk_level`, `policy`, `tool_policy`, `pii_analysis`, `safe_transcript`, `threats`.

Returns **HTTP 403** when `decision` is `block`, **HTTP 200** otherwise.

Response includes `scanned_transcript` — the primary speech prompt that was evaluated (from your ASR payload or server transcription).
When server transcription returns timing data, `asr.speech_segments` and `asr.word_timestamps` are preserved for audit evidence and review.

### VoiceGuard policies

Use `policy_id` to make tenant behaviour explicit:

- `voiceguard-default`: redacts PII from `safe_transcript`, blocks prompt/tool abuse, and disables tools for untrusted speech.
- `voiceguard-strict`: same as default, but blocks speech that contains PII or falls below the stricter ASR confidence threshold.
- `voiceguard-transcript-only`: for existing ASR pipelines where Koreshield should evaluate transcript text without requiring speaker verification.

You can also send an inline `audio_policy` object for tenant-specific overrides such as `redact_pii`, `block_on_pii`, `minimum_asr_confidence`, and `allow_untrusted_tool_calls`.

### Raw audio upload

```bash
curl -s -X POST https://api.koreshield.com/v1/audio/scan \
  -H "Authorization: Bearer <TOKEN>" \
  -F "audio=@call.wav" \
  -F "source_type=uploaded_audio" \
  -F "channel=user_speech" \
  -F "tools_available=lookup_customer,refund_payment"
```

Requires `OPENAI_API_KEY` on the Koreshield server for Whisper transcription, or send `transcript` / `audio_base64` in JSON instead.

For presigned recording links, send `audio_url` in JSON and enable `KORESHIELD_ALLOW_REMOTE_AUDIO_URLS=true` on the server. Koreshield only fetches HTTPS URLs, rejects private/local network targets, does not follow redirects, and enforces the same 25MB limit.

### Tool session auto-gate

When creating a runtime session, include `speech_context` (or top-level transcript fields). VoiceGuard runs before the session is returned:

```bash
curl -s -X POST https://api.koreshield.com/v1/tools/sessions \
  -H "Authorization: Bearer <TOKEN>" \
  -H "Content-Type: application/json" \
  -d '{
    "agent_id": "voice-agent",
    "allowed_tools": ["refund_payment"],
    "speech_context": {
      "transcript": "Summarise this call for the account notes.",
      "source_type": "live_microphone",
      "channel": "user_speech",
      "speaker_verified": true,
      "known_user": true,
      "asr": { "confidence": 0.96 }
    }
  }'
```

The response includes `voice_guard` and `speech_transcript`. Subsequent `/v1/tools/scan` calls with that `session_id` respect `tool_policy.allow_tool_calls`.

Scan history is available at `/v1/audio/scans` with the same list/get/delete/pack pattern as RAG scans.

## WebSocket

`/ws/events` provides real-time event streaming for dashboards and monitoring.

Auth via:
- Header: `Authorization: Bearer <jwt>`
- Cookie: `ks_access_token`

> **Note:** WebSocket endpoints are not part of the OpenAPI document. OpenAPI does not describe WebSocket handlers in this framework.

## Rate Limiting

All protected proxy endpoints use rate limits. The default limit can be adjusted in your dashboard settings (hosted) or deployment configuration (self-hosted).
