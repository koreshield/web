# Role-Based Access Control (RBAC)

KoreShield provides a built-in RBAC system for managing who can access what within your deployment. All RBAC endpoints require admin authentication.

## Built-in Roles

| Role | Description |
|---|---|
| `owner` | Full access, can manage all resources |
| `admin` | Administrative access, can manage users and config |
| `editor` | Can modify rules, policies, and configurations |
| `viewer` | Read-only access to dashboards and reports |
| `security_analyst` | Access to security events, alerts, and analytics |

## Permission Categories

Permissions are organized by category:

- **security**: view threats, manage rules, configure detection
- **management**: manage users, roles, API keys
- **analytics**: view dashboards, run reports
- **system**: server configuration, provider management

## API Endpoints

All RBAC endpoints live under `/v1/rbac`.

### Users

| Endpoint | Method | Description |
|---|---|---|
| `/v1/rbac/users` | GET | List users (search by name/email, filter by role) |
| `/v1/rbac/users/{user_id}` | GET | Get user details |
| `/v1/rbac/users` | POST | Create a new user |
| `/v1/rbac/users/{user_id}` | PUT | Update user (name, role, status) |
| `/v1/rbac/users/{user_id}` | DELETE | Delete a user |

### Roles

| Endpoint | Method | Description |
|---|---|---|
| `/v1/rbac/roles` | GET | List all roles |
| `/v1/rbac/roles/{role_id}` | GET | Get role details |
| `/v1/rbac/roles` | POST | Create a custom role |
| `/v1/rbac/roles/{role_id}` | PUT | Update role permissions |

### Permissions

| Endpoint | Method | Description |
|---|---|---|
| `/v1/rbac/permissions` | GET | List all permissions (filter by category) |

## User Statuses

- `active`: user can log in and access resources
- `inactive`: user account is disabled
- `pending`: user has been invited but has not accepted
