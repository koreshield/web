# Settings

This page mirrors `koreshield/config/config.yaml` and current auth hardening behavior.

## Server

```yaml
server:
  host: "0.0.0.0"
  port: 8000
```

## Logging

```yaml
logging:
  level: INFO
  json_logs: true
  container_mode: true
  file: logs/koreshield.log
```

## Security

```yaml
security:
  sensitivity: medium
  default_action: block
  features:
    sanitization: true
    detection: true
    policy_enforcement: true
```

## Providers

```yaml
providers:
  deepseek:
    enabled: true
    base_url: "https://api.deepseek.com/v1"
  openai:
    enabled: false
    base_url: "https://api.openai.com/v1"
  anthropic:
    enabled: false
    base_url: "https://api.anthropic.com/v1"
  gemini:
    enabled: false
    base_url: "https://generativelanguage.googleapis.com/v1beta"
  azure_openai:
    enabled: false
    base_url: "https://your-resource-name.openai.azure.com"
```

Set provider keys with env vars (`DEEPSEEK_API_KEY`, `OPENAI_API_KEY`, etc).

## Redis

```yaml
redis:
  enabled: true
  url: "${REDIS_URL}"
```

## JWT (Required)

```yaml
jwt:
  issuer: "koreshield-auth"
  audience: "koreshield-api"
```

Runtime requirements:

1. `JWT_ISSUER` and `JWT_AUDIENCE` must be configured.
2. Choose one key mode only:
- HS256: `JWT_SECRET` (minimum 32 chars)
- RS256: `JWT_PUBLIC_KEY` + `JWT_PRIVATE_KEY`
3. Do not set shared secret and RSA keys together.

## Alerting

`alerting.rules[].condition` now uses a safe parser/DSL (no `eval`). Keep conditions simple and explicit, for example:

- `attacks_detected > 10`
- `healthy_providers == 0`

## Management Config Endpoint

`GET /v1/management/config` behavior:

- disabled outside development-like environments (`404`)
- values are redacted for sensitive key names (`password`, `secret`, `token`, `api_key`, etc)
